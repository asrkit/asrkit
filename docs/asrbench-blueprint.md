## 一、对标分析(为什么有 asrbench 的位置)

### 1.1 HuggingFace open_asr_leaderboard(全球英文公开榜的标杆)

- **方法论一流,值得直接借鉴**:统一 normalizer 保证 WER 可比;WER + RTFx 双指标与 trade-off 散点图;固定硬件(HF Jobs / H200)+ 每家族 Docker 保证可复现;私有测试集防刷榜;评测成本公开标价。配套论文 arXiv:2510.06961。
- **架构笨重,是结构性的**:~20 个模型家族各写一套 `run_eval.py` + 16 份独立依赖 + 每家族一个 Docker。根因是"峰值性能公平"——榜单必须让每个模型跑自家最优推理路径,否则厂商质疑排名。每家族一套代码是这个承诺的代价。
- **盲区(实锤,非猜测)**:云端 10 家全是西方厂商(AssemblyAI/Azure/OpenAI/ElevenLabs/RevAI/Speechmatics 等),中国云厂零覆盖;无中文专项数据集(AISHELL/WenetSpeech/KeSpeech 均不在);sherpa-onnx 端侧生态缺席;多语是二等公民。

### 1.2 SpeechIO Leaderboard(中文先行者,论文必答题)

国内多年的中文 ASR 榜,评过主流中文云厂 API,有自己的中文归一化与 CER 口径。近年活跃度有限,基本不覆盖端侧 pull-即用生态。**任何论文/报告的 related work 必须正面回应"与 SpeechIO 的区别"**;开工前须把它的归一化规则与数据集口径研究透。

### 1.3 asrbench 的生态位(与两者都互补,不正面竞争)

四张独家牌,对标双方都做不出来:

| # | 独家维度 | 为什么他们做不了 |
|---|---|---|
| 1 | **端侧 + 云端同一把尺子**(SenseVoice/Paraformer/FireRed 端侧 vs 百炼/豆包/硅基流动云端) | HF 榜无中文无中国云厂;SpeechIO 基本只评云 API |
| 2 | **同模型跨引擎对比**(whisper 族 × faster-whisper/whispercpp/sherpa/transformers 四管道) | 引擎已编码在 asrkit model string 中,零成本枚举;他们每引擎一套 harness,口径不统一 |
| 3 | **成本维度**(¥/小时入榜,精度-速度-价格三维前沿) | 两家均无 |
| 4 | **薄架构**:一个 runner 吃全部模型 | 统一接口层(asrkit)是前提,他们没有 |

## 二、定位(先钉死,再动工)

### 2.1 一句话定位

> **测"开箱即得的部署现实",不测"实验室调优的峰值极限"。**

一条统一管道(asrkit)、同一台声明过的机器、`pip install asrkit && asrkit run`——你得到什么,榜上就是什么。这不是对"峰值公平"质疑的妥协,而是另一把更贴近用户的尺子:真实用户不会为每个模型手写 TensorRT 优化,他们要的就是"装上就跑,哪个又准又快又便宜"。

**预答质疑**:"你没发挥我模型的峰值速度" → 正确,峰值榜请看 open_asr_leaderboard;本榜测的是可 pip 安装的运行时里的开箱成绩,量化/引擎全部如实标注(见 4.4 诚实边界)。

### 2.2 战略顺序:中文先行,爬台阶,不一步登天

终点是"全球多语言多引擎评测平台",但正面在英文赛道与 HF(品牌 + H200 集群 + 社区)对轰不可取。台阶:

| 阶段 | 内容 | 独立价值 |
|---|---|---|
| **v0** | 中文全评:全部中文可用模型 × 中文数据集,CER 归一化做扎实,固定机器测 RTFx,附成本列 | 全球第一份中文端云横评;同时是 asrkit 的极限压力测试 |
| **v1** | 方法论固化 + 代码公开:normalizer 按语言插拔、数据集带 lang 属性、报告流水线自动化 | 平台骨架成型 |
| **v2** | 多语扩展:先加日/韩/粤等亚洲语言(同为西方榜盲区,Dolphin/SenseVoice 正好覆盖),英文"顺带能跑"不主打 | 错位竞争 |
| **v3** | 社区化:提交流程、防过拟合私有集、固定硬件 | 走到这步才谈"全球平台" |

每步独立可发布,走不到 v3 也不亏。**v0 = "用中文验证 asrkit 稳健性",它不是小目标,是全球平台的第一块地基。**

### 2.3 v0 的另一个身份:asrkit 的验收

一次中文横评 = 几十个模型 × 成百上千条音频,会把批量管道、adapter 缓存、云端重试、错误诚实、退出码契约全部打穿。**v0 的第一批产出物大概率是一串 asrkit 的 bug 与契约修订,这本身就是价值**(契约的最好测试是第一个真用户,expert-review C3)。

## 三、架构原则

```
asrbench/(独立仓库)
  datasets/     数据集注册表 —— lang 是【数据集的属性】,不是平台的属性
  normalizers/  按语言插拔 —— zh: 自建(壁垒);en: 借 open_asr_leaderboard(Apache-2.0)
  runner        唯一执行器 = 调 asrkit(消费 NDJSON / 退出码 / metrics 契约)
  metrics       WER / CER / RTFx / 成本
  report        散点图 + 表格 + 静态站
```

- **A1 · 语言即数据集属性**。即使 v0 只有中文,代码里也必须是 `dataset.lang → 选 normalizer/metric → 按 lang 筛模型`,绝不写死 "zh"。多语并存应是加数据集的加法,不是重构手术。
- **A2 · bench 保持薄,一切执行归 asrkit**。加新模型 = 给 asrkit 写 adapter/插件(entry-point 门已留)。飞轮:社区想上榜 → 先接入 asrkit → 评测平台反向给接口层导流。**依赖方向绝不反转。**
- **A3 · 模型圈定消费 asrkit 元数据**(W3a 修真后的 langs + W3b `list --lang`):中文数据集自动跳过 english-only 模型,不跑垃圾分。
- **A4 · 跨引擎"家族映射表"先放 bench 侧维护**(whisper-tiny 在四个引擎里的 id 对照)。若日后确实高频,再按老流程提议给 asrkit 元数据加 `family` 字段,不急着动核心。

## 四、评测方法论

### 4.1 指标

- **准确率**:中文 **CER**、英文 **WER**(按 dataset.lang 选择;混说/code-switch 后置)。
- **速度**:**RTFx**(音频时长 / 处理耗时),固定机器 + 公开声明口径。
- **成本**:云端按厂商牌价折 **¥/小时**;端侧标 0 + 硬件要求。三维画"精度-速度-价格"前沿。

### 4.2 归一化(论文的方法论章节,审稿人检验专业度之处)

- **中文(自建,核心壁垒)**:简繁、全半角、数字读法、标点剥离、ITN 口径——每条规则要有出处和理由,与 SpeechIO 口径的差异逐条说明。作为**独立开源组件**发布(可单独被引用)。
- **英文(不重造轮子)**:直接借 open_asr_leaderboard 的 normalizer(Apache-2.0),注明来源,保证与全球榜口径可对话。

### 4.3 可复现性

固定一台机器(型号/内存/GPU 公开声明)、锁依赖版本、全部代码与归一化器开源、逐条记录云端 API 调用日期(云端模型会静默升级,日期即版本)。

### 4.4 诚实边界(写进榜单页脚,不是小字)

1. **跨引擎 ≠ 严格同权重**:sherpa int8 / whispercpp ggml / faster-whisper CT2,同族不同量化。报告必须有"量化/格式"列——这正是"部署形态对比"的题中之义,不是缺陷。
2. **引擎对比矩阵不满**:能跨引擎的主要是 whisper 族;SenseVoice/Paraformer 等 sherpa 独占。"引擎横评"是子版块,不横贯全表。
3. **峰值未覆盖**:厂商自家 TensorRT 级调优不在本榜射程,声明"实验室极限 vs 部署现实"分界即可。
4. **云端结果带日期**:云厂模型非冻结产物,结论有保鲜期。

## 五、研究问题(v0 spec 的第一节,先于任何跑数)

**原则:数据不等于知识。表格是证据,主张才是论文;而主张不靠灵感,靠设计——问题设计好之后,任何答案都是发现。** 反面是唯一真正浪费的跑法:无对照结构的乱跑(模型随手挑、数据集不可比、引擎/量化/语言变量搅在一起)。

v0 预注册的研究问题(每个都不存在"无聊的答案"):

- **RQ1 端云差距**:中文端侧 SOTA(SenseVoice/Paraformer/FireRed)与云端(豆包/百炼/硅基流动)的 CER 差距是多少?差 1%(端侧够用了)与差 15%(云端不可替代)都是重磅结论。
- **RQ2 差距的条件性**:该差距在"干净朗读"与"口音/专有名词/嘈杂"数据集上如何变化?
- **RQ3 量化代价**:int8/ggml 量化对中文 CER 的损耗是多少?与英文文献里的结论一致吗?不同架构损耗一样吗?
- **RQ4 引擎效率**:同一 whisper 权重族,四条运行时管道的 RTFx 差多少?精度掉不掉?(直接指导用户装哪个 extra)
- **RQ5 价格-性能**:云端价格与准确率相关吗?"精度-速度-价格"前沿上谁在支配位?

跑数前用一个下午把 RQ 清单定稿——**那个下午就是"一堆日志"与"一篇论文"之间的全部距离。**

## 六、论文路径

### 6.1 门槛认知

"能上 arXiv"(无评审,谁都能挂)与"被会议接收"(需要 insight)是两个门槛。会议通行证 = 让领域知道了以前不知道的事;纯工程产物 + 大表会被 "just a leaderboard, no insight" 拒掉。第五节的 RQ 就是为跨过第二个门槛而设。

### 6.2 路径(按顺序)

1. **v0 跑完 → arXiv 技术报告**:零门槛先占坑、收社区反馈(open_asr_leaderboard 也是先挂 arXiv 改了 4 版)。
2. **正会投稿**:首选 **Interspeech / ICASSP**(语音圈主场,短文,benchmark 类接受度高);其次 NeurIPS Datasets & Benchmarks(分量重但卷)。
3. **兜底(即使"只有工具没有结论"也非零产出)**:**JOSS**(直接发表软件本身,评审代码/文档/测试——asrkit 的工程纪律是标准素材)、Interspeech Show & Tell / 系统 demo 轨、LREC 资源类(中文归一化器可单独作为"资源"发表)。

### 6.3 现实条件

独立作者无碍(benchmark 类拼可复现性与数据,不拼机构);成本结构轻(端侧免费,云端全量一轮约百元人民币量级,比 H200 账单低两个数量级);related work 必答 SpeechIO(见 1.2)。

### 6.4 价值排序(防止论文倒过来绑架项目)

> **asrkit 被人用 > asrbench 榜单被人信 > 论文被引用。**

论文是前两者的放大器,不是目的地。顺序永远是:东西有用 → 数据有权威 → 论文水到渠成。

## 七、明确不做(承接 roadmap / expert-review 既有清单)

- **不反向依赖**:asrkit 永不 import asrbench,不为 bench 加运行时耦合。
- **不追英文峰值榜**:英文赛道只"顺带能跑",不与 HF 榜正面对标。
- **不自研数据集(v0 阶段)**:先用公开集(AISHELL/WenetSpeech/KeSpeech 等,以 spec 定稿为准);自建私有防刷集是 v3 的事。
- **diarization / 强制对齐 / GUI**:沿用 asrkit 的不做清单,bench 侧同样不吞。

---

> 一句话:**用 asrkit 的统一接口把"20 份 harness"压成"1 个 runner",先在中文赛道做出全球没有的那份端云横评——榜单验收 asrkit,研究问题预注册好,论文只是水到渠成的副产品。**
